<!-- Auth challenge mechanism (MCP 2025-11-25): -->
When a tool call requires elevated auth, MCP server returns:
  { type: "error", code: "step_up_required", message: "..." }
or via WWW-Authenticate header on HTTP transports.
BFF receives this, checks the error code, and either:
  (a) triggers CIBA push to the user (out-of-band) if using CIBA flow
  (b) returns { requiresAuth: true } to the React SPA, which shows an inline login modal
After the user approves, the BFF retries the original tool call with a fresh elevated token.

2. **The three trigger scenarios in this demo**:
   - Amount threshold exceeded → CIBA push notification sent out-of-band
   - First use of a high-risk tool → PingOne Authorize gate check
   - Mid-flow session loss → inline login challenge in the browser
3. **The flow sequence** (prose + ASCII diagram):
   ```
   User clicks "Transfer $10,000" in agent
        ↓
   BFF POST /api/mcp/tool → mcpToolAuthorizationService
        ↓
   PingOne Authorize gate checks RAR/scopes
        ↓  [denied or step-up needed]
   BFF returns { error: "step_up_required", cibaRequired: true }
        ↓
   UI shows "Approval needed" state + polls BFF /api/ciba/status
        ↓  [user approves on device/email]
   BFF retries tool call with elevated token
        ↓
   Tool executes; result returned to agent
   ```
4. **Why this pattern matters** — agent operations consent while keeping the user in control; aligns with HITL best practices; token never escalated silently
