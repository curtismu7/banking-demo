**Section 2 — PingOne app configuration**
Step-by-step for each of the 3 OAuth applications:

*2.1 — Admin OIDC app (staff login + token exchange)*
- Create a new Application in PingOne → type: OIDC Web App
- Enable grant types: Authorization Code (PKCE required), Client Credentials
- Set redirect URI: `http://localhost:3001/api/auth/oauth/callback` (+ hosted URL when deploying)

- Note the Client ID and Client Secret → goes into PINGONE_AI_CORE_CLIENT_ID / PINGONE_AI_CORE_CLIENT_SECRET

*2.2 — End-user OIDC app (customer login)*
- Create a second Application → type: OIDC Web App
- Enable grant type: Authorization Code (PKCE only; no client credentials needed)
- Set redirect URI: `http://localhost:3001/api/auth/oauth/user/callback`
- Add scopes: `openid`, `profile`, `email`, `offline_access`, `banking:read`, `banking:write`, `banking:accounts:read`, `banking:transactions:read`, `banking:transactions:write`
- Note the Client ID and Client Secret → PINGONE_AI_CORE_USER_CLIENT_ID / PINGONE_AI_CORE_USER_CLIENT_SECRET

*2.3 — Worker app (Management API, optional for provisioning)*
- Create a third Application → type: Worker
- Assign PingOne environment role: "Identity Data Admin" or minimal `p1:read:user`, `p1:create:user` scopes depending on provisioning needs

*2.4 — Environment and resource server*

**Section 7 — Troubleshooting**
5 failure modes, each with: symptom, likely cause, fix:

1. **`invalid_client` at login** — Client ID not found in PingOne; or callback URI doesn't exactly match what's registered. Fix: compare PINGONE_AI_CORE_REDIRECT_URI in .env with the Redirect URI in PingOne Console exactly (no trailing slash difference).

2. **`invalid_scope` at login** — Scope not assigned to the OAuth app in PingOne. Fix: Go to PingOne Console → Applications → [app] → Resources tab → ensure all scopes in Section 2 are assigned. If using custom `banking:*` scopes, check they exist as a Custom Resource first.

3. **Sessions reset between requests** — SESSION_SECRET is too short or REDIS_URL is not set on a multi-instance deployment. Fix: generate a 32+ byte secret: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`. On Vercel: add Upstash Redis and set REDIS_URL.

4. **"Unsupported authentication method" during token exchange** — The BFF's `agentMcpTokenService.js` is using `client_secret_post` but the PingOne exchange client is configured as `client_secret_basic` (or vice versa). Fix: check the Token Endpoint Authentication Method in PingOne → the exchange client's Configuration tab; align AGENT_TOKEN_ENDPOINT_AUTH_METHOD in your .env.

5. **Dashboard shows no accounts (empty or 'undefined')** — Either `ENDUSER_AUDIENCE` in .env doesn't match the `aud` claim in the PingOne-issued access token, or the local SQLite accounts table was wiped on cold-start. Fix: decode the access token at jwt.io and compare its `aud` to ENDUSER_AUDIENCE. Restart the server if using cold-start mode to re-seed accounts.
