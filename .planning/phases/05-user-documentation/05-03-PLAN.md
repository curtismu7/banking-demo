**Section 2 — Why The BFF Holds All Tokens**

Explain the BFF Token Custodian Pattern:
- The browser never receives an access_token or refresh_token — only a session cookie
- The BFF is the sole token custodian: all tokens are stored server-side in the session (Upstash Redis on Vercel; better-sqlite3 or in-memory locally)
- This means a compromised browser session only exposes the session cookie — attacker still cannot replay the access_token directly
- All token exchange (RFC 8693), token validation, and MCP authentication happens inside the BFF where it is invisible to the client-side application
- Code pointer: `banking_api_server/services/oauthService.js` for token storage; `banking_api_server/services/agentMcpTokenService.js` for exchange

Include a simple "what is where" diagram in ASCII or a table:

| What | Where after login |
|------|------------------|
| access_token (user) | BFF session store (Redis / SQLite) |
| refresh_token | BFF session store |
| id_token | BFF session store (validated on receipt) |
| session_id | Browser cookie (httpOnly, SameSite=Lax) |
| MCP token | Derived via RFC 8693 exchange — BFF only, never stored in browser |

**Section 3 — Flow 1: Authorization Code + PKCE (User Login)**

Reference diagram: [`BX-Finance-AuthCode-PKCE-Flow.drawio`](./BX-Finance-AuthCode-PKCE-Flow.drawio)
RFC: [RFC 7636 — PKCE](https://datatracker.ietf.org/doc/html/rfc7636), [RFC 6749 — OAuth 2.0](https://datatracker.ietf.org/doc/html/rfc6749)

Subsections:

*3.1 What the user does*
"Click 'Log in as customer' → redirect to PingOne → enter credentials → land on /dashboard"

*3.2 Step-by-step narrative* (prose, follows the diagram steps)
Cover: BFF generates code_verifier/code_challenge; authorization request parameters; PingOne login; auth code callback; token exchange at BFF; session establishment.

*3.3 Token state table*

| Step | Action | access_token | refresh_token | id_token | Browser |
|------|--------|-------------|---------------|----------|---------|
| Before login | — | none | none | none | no tokens |
| After /authorize redirect | BFF builds PKCE params | none | none | none | state cookie |
| After PingOne callback | BFF exchanges auth code | held by BFF | held by BFF | validated by BFF | session cookie only |
| Dashboard loaded | BFF serves user data | held by BFF | held by BFF | held by BFF | session cookie only |

*3.4 PKCE rationale*
Why code_challenge/code_verifier matters: prevents auth code interception even if the redirect is intercepted. Per [RFC 9700 §2.1](https://datatracker.ietf.org/doc/html/rfc9700), PKCE is required for all Authorization Code flows.

**Section 4 — Flow 2: CIBA (Agent-Triggered Out-of-Band Approval)**

Reference diagram: [`BX-Finance-CIBA-Flow.drawio`](./BX-Finance-CIBA-Flow.drawio)
RFC: [CIBA Core spec](https://openid.net/specs/openid-client-initiated-backchannel-authentication-core-1_0.html)

Subsections:

*4.1 When CIBA is triggered*
"A high-value operation threshold is exceeded (default: transfers over $500), or a tool call requires explicit user consent. The AI agent itself cannot approve — a human must."

*4.2 Step-by-step narrative*
Cover: BFF POSTs to /bc-authorize with login_hint and binding_message; PingOne sends push to user's device; BFF polls /token with auth_req_id; user approves; BFF receives elevated token; agent tool call retried.

*4.3 Token state table*

| Step | Action | Token in BFF | Agent state |
|------|--------|-------------|-------------|
| Tool call initiated | Agent sends request | user_at (existing) | running |
| CIBA triggered | BFF calls /bc-authorize | user_at | waiting (cibaRequired) |
| Push sent | PingOne notifies user | user_at | polling |
| User approves | PingOne issues new token | user_at, elevated_at | polling |
| BFF receives elevated token | /token returns access_token | elevated_at | unblocked |
| Retry succeeds | Tool call completes | elevated_at | done |

*4.4 Code pointers*: `banking_api_server/routes/ciba.js` (bc-authorize + poll endpoints), CIBA threshold config in demo settings.

**Section 5 — Flow 3: Token Exchange (RFC 8693)**

Reference diagrams:
- [`BX-Finance-TokenExchange-Flow.drawio`](./BX-Finance-TokenExchange-Flow.drawio) — overview of both paths
- [`BX-Finance-1-Exchange-Delegated-Chain.drawio`](./BX-Finance-1-Exchange-Delegated-Chain.drawio) — 1-exchange detail
- [`BX-Finance-2-Exchange-Delegated-Chain.drawio`](./BX-Finance-2-Exchange-Delegated-Chain.drawio) — 2-exchange detail

RFC: [RFC 8693 — Token Exchange](https://datatracker.ietf.org/doc/html/rfc8693)

*5.1 Why token exchange*
When the AI agent calls an MCP tool, the user's session access_token cannot be passed directly to the MCP server — it was issued for the BFF's audience. The BFF performs a token exchange to get an MCP-scoped token with the narrowed audience.

*5.2 1-Exchange path*
Feature flag: `USE_AGENT_ACTOR_FOR_MCP=false`

Narrative: BFF takes user's access_token → calls PingOne /token with grant=urn:ietf:params:oauth:grant-type:token-exchange, subject_token=user_at, audience=banking_mcp_01 → receives MCP token (aud=banking_mcp_01) → passes to MCP server.

Token state table for 1-exchange:

| Step | Subject token | Result token | act claim |
|------|--------------|--------------|-----------|
| Before exchange | user_at (aud=BFF) | — | none |
| After exchange | user_at | mcp_at (aud=banking_mcp_01, sub=user) | none |
| MCP call | — | mcp_at sent as Bearer | none |

**Deep dive:** [PINGONE_MAY_ACT_ONE_TOKEN_EXCHANGE.md](./PINGONE_MAY_ACT_ONE_TOKEN_EXCHANGE.md) — full setup guide for the 1-exchange chain.

*5.3 2-Exchange path (agent actor)*
Feature flag: `USE_AGENT_ACTOR_FOR_MCP=true` + `ff_two_exchange_delegation=true`

Narrative: BFF first gets agent_actor_token via client credentials for the agent client. Then exchanges: subject_token=user_at + actor_token=agent_actor_token → MCP token with nested act claim proving the full delegation chain.

Token state table for 2-exchange:

| Step | Operation | Result | act claim |
|------|-----------|--------|-----------|
| Step 1 | BFF → PingOne: client_credentials for agent | agent_actor_at (sub=agent) | none |
| Step 2 | BFF → PingOne: token-exchange with user_at + agent_actor_at | mcp_at (sub=user, aud=banking_mcp_01) | { sub: agent } |
| MCP call | mcp_at sent as Bearer | — | { sub: agent } proves delegation |

Decoded act claim in the resulting token:
```json
{
  "sub": "user@example.com",
  "aud": "banking_mcp_01",
  "act": {
    "sub": "agent-client-id@client"
  }
}
```

**Deep dive:** [PINGONE_MAY_ACT_TWO_TOKEN_EXCHANGES.md](./PINGONE_MAY_ACT_TWO_TOKEN_EXCHANGES.md) — full setup guide for the 2-exchange chain, including PingOne Authorize policy configuration.

*5.4 Code pointer*: `banking_api_server/services/agentMcpTokenService.js` — the token exchange implementation.
