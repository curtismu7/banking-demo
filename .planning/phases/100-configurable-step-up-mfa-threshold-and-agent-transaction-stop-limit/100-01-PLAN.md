<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (UI) → BFF | Admin user must be authenticated to write security settings |
| BFF → PingOne | Step-up ACR values must match PingOne Sign-On Policy names exactly |
| End user → Agent | Agent enforces step-up threshold; must not be bypassable by client-side tampering |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-100-01 | Elevation of Privilege | SecuritySettings UI | mitigate | Admin role check on /api/admin/settings PUT; enforce `req.user.role === 'admin'` at route entry |
| T-100-02 | Tampering | transactions.js step-up gate | mitigate | Read threshold from server-side runtimeSettings (never trust client-provided threshold); enforce in BFF before transaction creation |
| T-100-03 | Information Disclosure | Error responses | mitigate | 428 response includes amount_threshold; do not expose internal settings or PingOne secret data |
| T-100-04 | Denial of Service | Step-up check loop | accept | Config reads are O(1); no risk of infinite loop or resource exhaustion |
</threat_model>
