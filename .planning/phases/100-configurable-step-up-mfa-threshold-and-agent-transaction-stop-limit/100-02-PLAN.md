<objective>
Implement enterprise-grade agent transaction governance. When an AI agent acts on behalf of a user (RFC 8693 delegated access), enforce a configurable limit on how many transactions or what cumulative value the agent can perform before requiring new explicit user approval.

Purpose: Address the security requirement from Phase 94 (Explicit HITL) by preventing surprise high-value or high-frequency agent activity without user re-approval.

Output:
- New `agentTransactionTracker.js` service for budget tracking
- Updated SecuritySettings UI with agent limit configs
- Updated transactions.js POST / with agent budget enforcement
- Unit tests for agent limit scenarios
- Runtime config for agent limit thresholds
</objective>

  <name>Task 1.1: Create agentTransactionTracker service</name>
  <files>banking_api_server/services/agentTransactionTracker.js</files>
  <action>
    Create a new service file that tracks delegated agent transactions and enforces limits.
    
    **Exports:**
    - `checkAgentTransactionBudget(req, amount, type)` → returns {ok: bool, remaining: {count, value}, error?: {status, json}}
      - Reads from runtimeSettings: agentTransactionLimit (count), agentTransactionValueLimit (value)
      - Tracks per-session via req.session (not per-global-agent, since agent acts on behalf of user)
      - Returns remaining budget if OK; error details if limit exceeded
    - `consumeAgentTransaction(req, amount, type)` → consumes one transaction from budget, updates session
    - `resetAgentBudget(req, newLimit?, newValueLimit?)` → Resets counter after user approves new agent action (integrate with Phase 94)
    
    **Session state structure:**
    ```javascript
    {
      agentTransactionTracker: {
        consumedCount: 0,
        consumedValue: 0,
        lastResetAt: Date,
        approvalTokenId: 'uuid from Phase 94 HITL', // Links budget reset to specific user approval
      }
    }
    ```
    
    **Logic:**
    - Skip tracking if NOT delegated access (req.user.isDelegated !== true) or if user is admin
    - If agentTransactionLimit = 0, skip count tracking (unlimited)
    - If agentTransactionValueLimit = 0, skip value tracking (unlimited)
    - On budget check: return {ok: false, status: 429, message: "Agent transaction limit reached..."}
    - Store service as banking_api_server/services/agentTransactionTracker.js
  </action>

  </verify>

  </done>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Agent (MCP client) → BFF | Delegated access must be proven by RFC 8693 `act` claim in JWT; budget is per-agent-session, not per-JWT |
| Admin (SecuritySettings) → BFF | Agent limit config must be written only by admin (enforced at /api/admin/settings route) |
| Session storage | Agent transaction budget stored in req.session (Redis on Vercel); survives across Lambda cold starts if session is preserved |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-100-05 | Elevation of Privilege | Agent limit config | mitigate | Admin-only endpoint (/api/admin/settings); enforced via requireRole middleware |
| T-100-06 | Tampering | Agent budget in session | mitigate | Session state is server-side (Redis); client cannot modify budget; validate req.user.isDelegated from JWT act claim |
| T-100-07 | Repudiation | Agent transaction activity | mitigate | Log each agent transaction with consumedCount, consumedValue, and budget remaining to audit trail (optional Phase 101+) |
| T-100-08 | Denial of Service | Budget check on every request | mitigate | Budget check is O(1); no loops; no resource exhaustion risk |
</threat_model>
