# Phase 100: Agent Token Exchange Flow

## Description
Implement the complete token exchange flow for the agent to obtain proper tokens for calling the MCP server. This includes two paths:

**Two-Exchange Path:**
1. User logs in via PingOne OAuth → obtains user token
2. Agent obtains its own token via client credentials to PingOne
3. Token exchange: user token + agent token → MCP token with `act` claim
4. MCP server receives and validates the MCP token

**Single-Exchange Path:**
1. User logs in via PingOne OAuth → obtains user token
2. User token (authz) token exchange → Agent token
3. MCP server receives the agent token

## Dependencies
- Phase 99: langgraph-upgrade (LangGraph agent implementation)
- Phase 6: token-exchange-fix (RFC 8693 token exchange infrastructure)

## Deliverables
- Agent token acquisition via client credentials
- Two-exchange token exchange implementation (user + agent → MCP)
- Single-exchange token exchange implementation (user → agent → MCP)
- MCP server token validation for both paths
- Token chain event tracking for agent operations
- Error handling for token exchange failures
- Session storage of agent tokens
- Configuration for exchange mode selection

## Estimated Duration
2-3 days

## Success Criteria
- Agent can obtain its own token via client credentials when needed
- Two-exchange flow works: user token + agent token → MCP token with `act` claim
- Single-exchange flow works: user token → agent token → MCP server
- MCP server correctly validates tokens from both paths
- Token chain display shows the correct exchange path used
- Agent tool calls work without authentication errors in both modes
- Session persists agent tokens correctly using SQLite (local) or Redis (Vercel)
