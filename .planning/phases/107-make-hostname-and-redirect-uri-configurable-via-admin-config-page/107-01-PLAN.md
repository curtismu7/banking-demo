Create a new service that manages hostname configuration. Requirements:
- Module exports: `getConfiguredHostname()`, `setConfiguredHostname(hostname)`, `InvalidHostnameError` class
- `getConfiguredHostname()`: Returns currently configured hostname, defaults to `https://api.pingdemo.com:4000` if not yet configured
- `setConfiguredHostname(hostname)`: Validates and saves hostname; throws `InvalidHostnameError` if invalid
- Validation rules for hostname:
  - Must include protocol: `https://` or `http://`
  - Must include host (domain or localhost)
  - May include port (e.g., `:4000`, `:3000`)
  - Must match regex: `^https?:\/\/[a-zA-Z0-9.-]+(:\d{1,5})?$`
  - Examples of valid: `https://api.pingdemo.com:4000`, `http://localhost:3000`, `https://staging.app.com`, `https://localhost:4000`
  - Examples of invalid: `api.pingdemo.com` (missing protocol), `http://localhost:` (missing port), `ftp://localhost` (wrong protocol), `http://localhost:99999` (port out of range)
- Store in memory using a module-level `let` variable (initialize from configStore on first read)
- Persist to `configStore` using key `'CONFIGURED_HOSTNAME'` (follows existing pattern)
- All hostname changes logged via `logger.info()` with LOG_CATEGORIES.CONFIG_UPDATE
- Error logging via `logger.error()` for validation failures
- Use CommonJS (`require`/`module.exports`), 2-space indentation, camelCase function names

  <action>
Add two endpoints to `banking_api_server/routes/admin.js`:

**Endpoint 1: GET /api/admin/config/hostname**
- No authentication required (per STRUCTURE.md: /api/admin/config/* is unprotected)
- Returns JSON: `{ hostname: "https://api.pingdemo.com:4000" }`
- Status: 200 on success
- Error handling: 500 if service fails

**Endpoint 2: PUT /api/admin/config/hostname**
- No authentication required (consistency with GET)
- Request body: `{ hostname: "https://staging.app.com:4000" }`
- Calls `configHostnameService.setConfiguredHostname(hostname)`
- Returns JSON: `{ hostname: "https://staging.app.com:4000", updated: true }`
- Status: 200 on success, 400 if validation fails (catch InvalidHostnameError), 500 on other errors
- 400 response body: `{ error: "Invalid hostname format", details: "..." }`

**Implementation pattern (follow existing admin.js code style):**
- Require configHostnameService at top with: `const configHostnameService = require('../services/configHostnameService');`
- Use `router.get()` and `router.put()` 
- Error handling: try/catch InvalidHostnameError and return 400 with error message

**Example handler:**
```javascript
router.get('/config/hostname', (req, res) => {
  try {
    const hostname = configHostnameService.getConfiguredHostname();
    res.json({ hostname });
  } catch (error) {
    logger.error('Failed to get hostname config', { error });
    res.status(500).json({ error: 'Failed to get hostname config' });
  }
});

router.put('/config/hostname', (req, res) => {
  try {
    const { hostname } = req.body;
    if (!hostname) {
      return res.status(400).json({ error: 'hostname is required' });
    }
    configHostnameService.setConfiguredHostname(hostname);
    res.json({ hostname, updated: true });
  } catch (error) {
    if (error instanceof configHostnameService.InvalidHostnameError) {
      return res.status(400).json({ error: error.message });
    }
    logger.error('Failed to set hostname config', { error });
    res.status(500).json({ error: 'Failed to set hostname config' });
  }
});
```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-107-01 | Injection | configHostnameService | mitigate | Regex validation in `setConfiguredHostname(hostname)` — only `https?://` schemes, alphanumeric + dot + dash for domain, numeric ports 1-65535. No shell commands, no path traversal. |
| T-107-02 | Spoofing | GET /api/admin/config/hostname | accept | Low risk — hostname config is not authentication-related, discoverable from any client, not security-sensitive per D-05. |
| T-107-03 | Tampering | PUT /api/admin/config/hostname | mitigate | No authentication required per phase scope (admin page accessible after auth). HTTPS enforced by browser redirect requirement. Server-side validation prevents invalid URIs from causing OAuth failures. |
| T-107-04 | Information Disclosure | GET hostname | accept | Low risk — hostname is deployment topology info, not sensitive credentials. Already exposed in .env files and error messages. |
| T-107-05 | Denial of Service | PUT /api/admin/config/hostname | mitigate | API is unprotected (no auth required), but validation on hostname format (max length ~256 chars) and port range (1-65535) prevents abuse. No rate limiting added (falls to broader API rate limits if configured). |
| T-107-06 | Elevation of Privilege | /api/admin/config/* | mitigate | Endpoints unprotected per spec (admin-only access managed by frontend AdminRoute guard, not BFF). Setting invalid hostname results in OAuth failures, not privilege escalation. |
