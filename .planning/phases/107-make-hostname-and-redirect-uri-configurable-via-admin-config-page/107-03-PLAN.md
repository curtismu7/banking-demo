<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Configured Hostname → OAuth Request | BFF sends user-configured hostname in OAuth authorization URL to PingOne; must match registered redirect URI in PingOne. |
| PingOne Callback → BFF Validation | PingOne redirects to redirect_uri; BFF validates hostname matches configured value before processing callback. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-107-I-01 | Tampering | OAuth redirect_uri parameter | mitigate | BFF constructs redirect_uri from validated configured hostname (validated in Plan 01 Task 1). PingOne server-side checks that redirect_uri in callback matches registered values. If admin configures wrong hostname, OAuth fails (no bypass). |
| T-107-I-02 | Spoofing | OAuth callback host | mitigate | Redirect URI must be HTTPS (enforced by Phase 107 CONTEXT.md D-07). Admin misconfiguring HTTP hostname on production will fail PingOne's HTTPS requirement. BFF validates callback against configured hostname before processing. |
| T-107-I-03 | Elevation of Privilege | Admin changes hostname | accept | Only admin users can change hostname (AdminRoute guard at frontend, not BFF level per spec). If compromised, attacker could redirect logins to malicious host — falls to PingOne's URL validation. Audit log recommended (deferred to future phase). |
| T-107-I-04 | Denial of Service | PUT /api/admin/config/hostname | mitigate | No auth required per spec. Invalid hostnames rejected by validation, so admin can't break OAuth. Valid but unreachable hostnames (e.g., typo in domain) just cause login failures, not DoS. |

</threat_model>
