  <name>Task 2: Wire 504 errors in bankingAgentService.js fetch calls</name>
  <files>banking_api_ui/src/services/bankingAgentService.js</files>
  <action>
Update API error handling in bankingAgentService to trigger restart notifications on 504/timeout.

**Pattern to follow:**

Find all `fetch()` calls in bankingAgentService.js and wrap them with 504 detection. Current pattern (line ~50–80):

```jsx
try {
  const res = await fetch(url, options);
  if (!res.ok) {
    if (res.status === 504) {
      // NEW: Signal restart notification
      // The notification service is already monitoring, so this is optional logging/tracking
      console.warn('[BankingAgent] 504 from server, restart notification should be showing');
    }
    // Existing error handling for non-504 errors
  }
  return res.json();
} catch (err) {
  if (err.name === 'AbortError' || err.message.includes('timeout')) {
    // NEW: Timeout also triggers restart notification
    console.warn('[BankingAgent] Connection timeout, restart notification should be showing');
  }
  throw err;
}
```

**Key updates:**

1. **Check status 504:** After `const res = await fetch()`, check `if (res.status === 504)` → log a warning (the service is already handling via global wrapper, this is just local logging for context)

2. **Catch timeouts:** In the catch block, detect `err.name === 'AbortError'` or error message contains "timeout" → log warning

3. **Don't block existing flow:** These updates are instrumentation only. Existing error handling continues as-is; the service layer handles the actual modal display.

4. **Scope:** Update all major API call sites:
   - `/api/mcp/tools/call` (tool invocations)
   - `/api/mcp/tools/list` (capabilities fetch)
   - `/api/banking/accounts` (account data)
   - `/api/admin/*` (admin operations)
   - Any other endpoints called from this service

5. **Minimal diff:** Do NOT refactor existing error handling. Only add `if (res.status === 504)` checks and timeout detection. Keep existing `.catch()` blocks and error messages.

**No external dependency changes** — use existing error handling patterns.
