<objective>
Implement conditional authentication flow for banking transactions where logged-in users only require MFA for high-value transactions, while non-logged-in users require both login and MFA. The implementation must respect the existing step-up MFA threshold configuration and provide clear UI messaging about which authentication step is required.

Purpose: Enable frictionless banking for authenticated users while maintaining security for high-value transactions, and ensure unauthenticated users complete full authentication before any banking actions.

Output:
- Conditional auth gate in transactions.js that checks session before MFA
- Session-aware error responses (401 for login required, 428 for MFA required)
- UI components that display appropriate messaging based on auth state
- No regression in existing step-up MFA logic
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (UI) → BFF | Session cookie must be validated server-side |
| BFF → Session Store | Session data must be read from Redis/local store, not client headers |
| Agent → BFF | Agent must respect same auth gates as direct UI calls |
| Step-up Gate → PingOne | MFA flow must only trigger for logged-in users above threshold |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-122-01 | Elevation of Privilege | Session check bypass | mitigate | Server-side session validation only; never trust client-side auth state |
| T-122-02 | Tampering | Threshold manipulation | mitigate | Read threshold from server-side runtimeSettings; never accept from client |
| T-122-03 | Information Disclosure | Error responses | mitigate | 401 response includes login_url only; do not expose internal session data |
| T-122-04 | Spoofing | Agent bypass | mitigate | Agent uses same API endpoints; subject to same session/MFA gates |
</threat_model>
