<objective>
Fix two UI-side breakage points where the token chain stays stuck on placeholder:

1. **UserDashboard never calls `useCurrentUserTokenEvent`** — only `Dashboard.js` (admin) calls it. Customer users on `/dashboard` never seed the chain from their session token.
2. **`fetchSessionPreview` stale-skip guard** — skips re-fetch if `ctx.events.length > 0`, even after a new login (`userAuthenticated` event). This means after re-authenticating mid-session, the chain shows stale events.
3. **No visible error state** — when neither live events nor session preview resolves, the component shows generic "placeholder" cards with no explanation.

Purpose: After this plan, customer dashboard always seeds the chain; re-auth triggers a fresh fetch; empty state tells the user what happened.
Output: `useCurrentUserTokenEvent` added to UserDashboard, stale guard fixed, CSS `tcd-empty-state` added.
</objective>

**Fix 1 — stale fetch guard in `fetchSessionPreview`:**

The guard `if (ctx && ctx.events.length > 0) return;` prevents re-fetching after re-auth because stale live events are still present.

Change the guard to only skip if we already have a session preview (not if live events exist), and always re-fetch when triggered by `userAuthenticated`:

Replace:
```js
const fetchSessionPreview = useCallback(async () => {
  if (ctx && ctx.events.length > 0) return; // live data present — skip
```

With:
```js
const fetchSessionPreview = useCallback(async (force = false) => {
  if (!force && Array.isArray(sessionPreviewEvents) && sessionPreviewEvents.length > 0) return; // already loaded — skip
```

Then in the `userAuthenticated` event handler, call with `force = true`:
```js
const onAuth = () => {
  setSessionPreviewEvents(null);
  void fetchSessionPreview(true); // force refresh on re-auth
};
```

The `sessionPreviewEvents` ref in the useCallback closure will be stale — use a ref to track it OR just remove the skip-if-present guard entirely and rely on the `userAuthenticated` pattern.
