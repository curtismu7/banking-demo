## PingOne Test Page educational additions:

### 1. Claim tooltips in DecodedTokenPanel
Add optional `claimMeta` prop — a map of claim name → `{ label, description }`. When provided, each claim row shows an ℹ️ icon with a tooltip. Default meta covers standard OAuth/OIDC claims:

| Claim | Label | Description |
|---|---|---|
| `iss` | Issuer | The PingOne Authorization Server that issued this token |
| `sub` | Subject | The user or client this token was issued for |
| `aud` | Audience | Which API / resource server can accept this token |
| `exp` | Expires At | Unix timestamp when this token becomes invalid |
| `iat` | Issued At | When this token was minted |
| `scope` | Scopes | Permissions granted to the bearer |
| `client_id` | Client | OAuth application that requested this token |
| `act` | Actor | Who is acting on behalf of the subject (RFC 8693 delegation) |
| `may_act` | May Act | Who is permitted to act on behalf of this subject |
| `jti` | Token ID | Unique identifier for this token (for revocation tracking) |
| `acr` | Auth Context | How the user authenticated (e.g. urn:pingone:mfa) |
| `amr` | Auth Methods | List of authentication methods used |
| `env` | Environment | PingOne environment ID this token belongs to |

### 2. Token Lineage section for exchange cards
Below the DecodedTokenPanel on exchange cards, add a collapsible "Token Lineage ▸" section:
- Shows two columns: Input Token Claims | Output Token Claims
- Highlights CHANGED claims (e.g. aud changes from PingOne API → MCP server URI)
- Highlights ADDED claims (e.g. `act` appears in output)
- Highlights REMOVED claims
- Color: green=added, amber=changed, grey=same, red=removed

### 3. What is Happening panels (PingOne Test)
Each test card gets a collapsible "What is Happening ▸" panel (rendered above SectionApiCalls):
- Worker Token: "BFF calls PingOne /as/token with client_credentials grant. The worker token lets the BFF call PingOne Management API on behalf of the application."
- Auth Code Token: "User completed Authorization Code + PKCE flow. This token proves who the user is."
- Agent Token: "BFF uses a second OAuth client (MCP Token Exchanger) to get a client_credentials token. This proves the AI agent's identity."
- Single Exchange: "BFF sends the user's token to PingOne /as/token with grant_type=token-exchange. PingOne mints a new token scoped to the MCP server."
- Double Exchange: "BFF sends both user token AND agent token. PingOne mints a delegated token with act claim showing the agent acted on behalf of the user."

## MFA Test Page educational additions:

### 4. Per-section "What is Happening" panel
Each of the 5 MFA sections gets a collapsible info panel:
- SMS OTP: "PingOne sends OTP via SMS using DaVinci flow. Step 1: initiate (gets DA ID). Step 2: submit OTP to verify."
- Email OTP: "Same as SMS but delivery channel is email. PingOne calls the configured email service."
- FIDO2: "Browser generates a cryptographic key pair. Public key stored in PingOne. Private key stays in device secure enclave. No shared secret."
- Device Management: "Lists all registered MFA devices for the user. Devices have type, nickname, status, and createdAt."
- Enrollment: "Registers a new MFA device. Email enrollment creates a Device record of type EMAIL. FIDO2 enrollment requires browser WebAuthn API."

### 5. DA response field labels (MFA page)
When a DA response is shown (smsDaId, emailDaId, fidoDaId), render a labeled card:
- `daId`: "Device Authorization ID — reference for this MFA challenge session"
- `status`: "PENDING → OTP sent. COMPLETED → user verified. EXPIRED → time window passed."
- `expiresAt`: "Challenge expires at this time. User must enter OTP before then."
- `type`: "Which MFA method PingOne is expecting (SMS_OTP, EMAIL_OTP, FIDO2)"
