**Implementation pattern:**

Add a state to track pending auth:
```jsx
const [authPending, setAuthPending] = useState(false);
```

In each handle function (handleTransfer, handleDeposit, handleWithdraw):
```jsx
const handleTransfer = async (e) => {
  e.preventDefault();
  setAuthPending(true);
  try {
    // ... existing logic ...
    const result = await apiClient.post('/api/banking/transfer', payload);
    // ... success handling ...
  } catch (err) {
    // ... error handling ...
  } finally {
    setAuthPending(false);
  }
};
```

Update the button JSX to show pending state:
```jsx
<button 
  type="submit" 
  className="transfer-btn"
  disabled={!isLoggedIn || authPending}
  title={!isLoggedIn ? "Log in to transfer funds" : ""}
>
  {authPending ? (
    <>
      <span className="spinner-inline"></span>
      Verify your identity...
    </>
  ) : (
    'Transfer'
  )}
</button>
```

Repeat for deposit and withdraw handlers/buttons.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User action → disabled attribute | Client-side disable is for UX only; server-side auth check is primary security |
| Toast/error display | Error messages from API must not leak sensitive data (handled in existing error handler) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-142-04 | Elevation of Privilege | Client bypasses disabled attribute via DevTools | mitigate | Server-side auth check in API route (req.user validation); disabled button is UX only |
| T-142-05 | Denial of Service | Rapid button clicks during pending state | mitigate | Button disabled during authPending; handler uses try/finally to always reset flag |
| T-142-06 | Information Disclosure | "Verify your identity" message reveals auth flow | accept | Message is user-facing, non-sensitive; no data leak |

</threat_model>
