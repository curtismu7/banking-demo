<task type="auto">
  <name>Task 2: Create dual-layer error handler service with user-friendly + technical + recovery steps</name>
  <files>banking_api_ui/src/services/errorHandler.js</files>
  <action>
Create errorHandler.js service to standardize error formatting for agent operations.

Two-layer error display:
- Layer 1 (User-friendly): Concise, non-technical message shown first. E.g., "Agent couldn't complete this transfer. Check your token configuration or try manually."
- Layer 2 (Technical): Expandable/collapsible section showing technical detail. E.g., "(Token exchange error: insufficient_scope — verify agent app has banking scope)"
- Layer 3 (Recovery): Link to troubleshooting doc or inline "How to fix this" steps. E.g., "Troubleshooting agent actions → https://docs/agent-errors"

Implement functions:
- formatAgentError(error: {code, message, details}): Promise<{userMessage, technicalDetail, recoverySteps}>
- getErrorRecoverySteps(errorCode: string): {title, steps: [], docLink?}

Error categories to handle:
- Token exchange errors (insufficient_scope, invalid_grant, auth failed)
- MCP tool errors (tool not found, invalid params, tool call failed)
- Banking API errors (insufficient funds, account not found, limit exceeded)
- Network/timeout errors
- Permission/authorization errors

Example usage in BankingAgent:
```javascript
try {
  await agentMcpTokenService.resolveMcpToken();
  // ...
} catch (err) {
  const formatted = await errorHandler.formatAgentError(err);
  showErrorModal({
    title: "Agent Action Failed",
    userMessage: formatted.userMessage,
    technicalDetail: formatted.technicalDetail,
    recoverySteps: formatted.recoverySteps,
    docLink: formatted.docLink
  });
}
```

Rationale: Per D-07, users need to understand why agent failed + how to fix it. Dual layers prevent cognitive overload while providing technical depth for debugging.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → BFF | User is authenticated; approval decision is local state only (no external validation) |
| Agent request → MCP Token Service | Agent must provide valid user ID token before exchange; validation at BFF only |
| Error display → User | Error messages must not leak PII, internal paths, or credentials |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-143-01 | Tampering | ApprovalModal / user decision | Mitigate | Approval state stored in React Context (client-side only); no persistence across page refresh. User approval sent to backend in token exchange request only. |
| T-143-02 | Information Disclosure | Error messages display technical + user layer | Mitigate | Error messages show technical details (sufficient for debugging) but NEVER include: session tokens, PII, system paths, database queries. Audit error formatting in formatAgentError(). |
| T-143-03 | Spoofing | Agent claims to execute on behalf of user | Mitigate | Agent identity verified via MCP token `act` claim at backend (Phase 144). Frontend cannot spoof agent; approval modal just captures user intent. |
| T-143-04 | Repudiation | User claims didn't approve transaction | Accept | User approval is local state; no backend audit trail. Rationale: agent operates within approved token scope; transaction itself is auditable. This phase adds UX consent, not audit trail. |
| T-143-05 | Denial of Service | User spams "Reject" → agent retries forever | Accept | Rate limiting not in scope for Phase 143. Agent retries are user-controlled via chat interface. |
| T-143-06 | Elevation of Privilege | Agent bypasses threshold by splitting transaction | Mitigate | Threshold is per-transaction at chat level, not backend enforcement. Backend cannot enforce (agent controls params). Note as limitation: "Threshold is UX guidance only; backend should add amount limits per agent role (future phase)." |

</threat_model>
