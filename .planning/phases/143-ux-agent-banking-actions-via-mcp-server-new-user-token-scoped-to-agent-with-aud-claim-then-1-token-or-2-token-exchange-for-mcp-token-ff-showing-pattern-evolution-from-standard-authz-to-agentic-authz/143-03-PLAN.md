Add feature flag FF_AGENT_2_TOKEN_EXCHANGE to control default token exchange path (1-token vs 2-token).

Frontend (configStore.js):
- Add getFeatureFlag('AGENT_2_TOKEN_EXCHANGE'): boolean
- Returns true for 2-token, false for 1-token (default)
- Read from BFF /api/config/feature-flags endpoint
- Call setFeatureFlag('AGENT_2_TOKEN_EXCHANGE', boolean) to toggle via POST /api/config/set-flag

Backend (config.js routes):
- GET /api/config/feature-flags → returns { AGENT_2_TOKEN_EXCHANGE: boolean, ... }
- POST /api/config/set-flag → { flag: 'AGENT_2_TOKEN_EXCHANGE', value: boolean } 
  - Updates process.env and in-memory store
  - Returns { success: true, flag, newValue }
- Admin-only access (auth middleware)

Admin Config Page UI:
- Existing config page now has section: "Token Exchange Path"
- Toggle: "Use 2-token exchange (RFC 8693 delegation mode)"
- Display current path: "Active path: 1-exchange" or "Active path: 2-exchange"
- On toggle, POST to /api/config/set-flag and update local state
- Next agent action uses the new path

Data flow:
1. Admin toggles FF on config page
2. POST /api/config/set-flag called with true/false
3. BFF stores FF in process.env
4. Next agent call to agentMcpTokenService checks getFeatureFlag(); uses new path
5. TokenChainDisplay shows new exchange path (1-exchange or 2-exchange)

Rationale: Per D-03, FF allows demo of both token exchange patterns live. Presenter can switch during conference talk to show both RFC 8693 paths in action.
