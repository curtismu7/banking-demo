<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → BFF | Scope names in to be validated tokens; injection flag must be authenticated admin-only |
| BFF → PingOne | Resource server must exist in PingOne or fallback to injection (handled by flag) |
| BFF → Database | Scope enforcement in code (not trusting scope from untrusted external source) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-146-01 | Tampering | Scope injection flag (`ff_inject_scopes`) | Mitigate | Feature flag only writable by admin (e.g., `/api/admin/feature-flags` requires `banking:admin` scope) |
| T-146-02 | Spoofing | Fake resource server in PingOne | Mitigate | During setup, verify resource server exists via Management API before enabling real scopes; log creation/verification |
| T-146-03 | Repudiation | Scope injection not audited | Mitigate | Log all scope injections to tokenEvents and exchange audit log (fire-and-forget entry in Redis) |
| T-146-04 | Information Disclosure | Scope names leaked in error responses | Accept | Scope names are not secrets; they're public OAuth scope names; error responses include scope names per OAuth spec |

</threat_model>
