**Step 0 — Read before writing:**
Read `banking_api_ui/src/components/DemoData.js` to understand:
1. How other sections (e.g., user profile, transaction settings) are structured and saved
2. What BFF endpoint is called to persist demo data (likely `POST /api/demo/scenario`
   or `PATCH /api/demo/...`)
3. How account-level data is currently loaded (does it call `GET /api/accounts/my`
   or use a demo scenario store?)
4. The state management pattern (local state + save button? Or auto-save on change?)

Do not write any code until you understand this pattern.

**Then implement:**
Add a new section titled "Account Profile Fields" after the existing account-related
sections (or at the end of the page — follow existing layout order).

Section structure (per account — render one sub-section per account):
- Section heading: "Account Profile Fields" with a lock badge "🔒 Some fields require banking:sensitive:read"
- Per-account sub-heading: "Checking Account" / "Savings Account" (use account.name)
- For EACH of these fields, render an editable text input using the existing DemoData
  field input pattern:

  Public fields (no visibility toggle needed):
  - `swiftCode` — label "SWIFT Code", default "CHASUS33"
  - `iban` — label "IBAN", default "US12CHAS..."
  - `branchName` — label "Branch Name", default "BX Finance Main Branch"
  - `branchCode` — label "Branch Code", default "001"
  - `openedDate` — label "Opened Date", type date or text, default "2022-01-15"
  - `accountHolderName` — label "Account Holder Name"

  Sensitive fields (render with a "🔒 Sensitive" badge + visibility toggle):
  - `routingNumber` — label "Routing Number 🔒", with "Include in response" toggle
  - `accountNumberFull` — label "Full Account Number 🔒", with "Include in response" toggle

The "Include in response" toggle is a checkbox. When unchecked, the field is stored but
omitted from `GET /api/accounts/my` (backed by a per-field `include_*` flag in the
demo scenario store). This is a demo-layer concern, not a security concern — the real
security is the scope+PAZ gate in Plan 02.

**Save mechanism:** Follow the identical pattern already used in DemoData.js for other
fields. Read the file first to see if it uses a "Save Changes" button per section, an
auto-save onChange, or a top-level save. Match it exactly — do not introduce a new save
mechanism.

**Load mechanism:** On component mount, if DemoData.js already loads account data from
the store, extend that load. If not, fetch from `GET /api/accounts/my` and populate the
form fields with the current values (accountNumberFull not included in that response, so
leave it blank initially, populated only if demo scenario data exists).
