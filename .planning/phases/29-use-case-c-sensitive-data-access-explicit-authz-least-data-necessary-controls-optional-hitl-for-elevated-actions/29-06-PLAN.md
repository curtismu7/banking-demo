const tabs = [

**Tab 1: Least-Data Principle** content sections:
- "What is the Least-Data Principle?" — Agents should request and receive ONLY the data
  they need for the task at hand. Full account numbers, routing numbers, or SSNs should
  never be in an agent context unless explicitly required.
- "Field-Level Scopes" — Instead of `banking:read` (all data), use granular scopes like
  `banking:accounts:read` and `banking:sensitive:read`. Each scope maps to a specific
  data tier.
- "Masking patterns" — Always return masked values by default (`****5678`). Only return
  full values after explicit authorization (scope + policy + consent).
- "Why agents shouldn't get full PII" — LLM context windows get logged, cached, and may
  be used in training. Minimizing PII scope limits exposure surface.
- Include a code snippet showing the two response shapes:
  ```json
  // Default (no sensitive scope):
  { "accountNumber": "****5678", "routingNumber": null }

  // After banking:sensitive:read + PAZ + consent:
  { "accountNumber": "****5678", "accountNumberFull": "010123456789", "routingNumber": "026073150" }
  ```

**Tab 2: RAR / RFC 9396 Selective Disclosure** content sections:
- "RAR in one sentence" — Rich Authorization Requests let a client declare WHAT it
  wants to do with the data, not just WHICH scopes. The authorization server and policy
  engine can make finer-grained decisions.
- "banking:sensitive:read as a RAR scope" — Show a concrete authorization_details object:
  ```json
  {
    "type": "banking_data_access",
    "actions": ["read"],
    "datatypes": ["account_number_full", "routing_number"],
    "identifier": "account-xyz"
  }
  ```
- "Why this matters" — A blanket `banking:read` scope gives the agent everything, forever.
  RAR lets the user approve EXACTLY which fields are exposed, to which agent, for how long.
- "How this demo uses it" — The `banking:sensitive:read` scope is a simplified RAR-adjacent
  pattern. A full RAR implementation (RFC 9396 §7) would pass `authorization_details` in
  the token request for even finer control.
- Cross-reference: "See also: RARPanel.js (general RAR introduction)" — note this panel
  focuses on the banking data access use-case specifically.

Color accent: Use a teal/blue-green scheme (lock + information tone) consistent with
the security-themed panels in this project.
