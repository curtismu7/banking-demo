  <name>Task 1: Diagnose and fix POST /api/mcp/tool 400 error (D-16)</name>
  <files>banking_api_server/server.js</files>
  <action>
    First, reproduce the 400 via curl with a running server:
    ```
    curl -sv -X POST http://localhost:3001/api/mcp/tool \
      -H 'Content-Type: application/json' \
      -d '{"tool":"get_my_accounts","params":{}}'
    ```
    The error response at line 1020 says "Received body keys: [...]".

    Root cause analysis:
    - If body keys are empty ([]), the body was not parsed. This happens when the
      global express.json() already consumed the stream before the route-level one ran.
    - The raw buffer fallback at lines 998-1010 guards with `req.readable`. After the body
      parser reads the stream, `req.readable` may still be truthy but the stream yields no more
      chunks — so the raw read returns nothing and parsedBody.tool stays undefined.

    Fix: in server.js, change the raw buffer fallback guard condition from:
      `if (!parsedBody.tool && req.readable)`
    to:
      `if (!parsedBody.tool && req.readableLength > 0)`

    This ensures the fallback only attempts to re-read when there are actually unread bytes.

    Additionally, add one diagnostic log line just before the 400 return (only fires on failure),
    so Vercel logs capture body keys:
      `console.warn('[/api/mcp/tool] 400: body keys=[%s] readableLength=%d', Object.keys(parsedBody).join(','), req.readableLength);`

    If the issue cannot be reproduced locally (might be Vercel-only cold-start), apply the
    guard fix anyway as a defensive improvement — it's clearly more correct.

    Do NOT change the /api/mcp/inspector/invoke route in mcpInspector.js — it already applies
    express.json() inline correctly and is a separate route with its own guard.
  </action>
  <verify>
    curl -X POST http://localhost:3001/api/mcp/tool \
      -H 'Content-Type: application/json' \
      -d '{"tool":"get_my_accounts","params":{}}' \
      → Response status must NOT be 400 (expected: 401 without session, or 200 with auth)
  </verify>
