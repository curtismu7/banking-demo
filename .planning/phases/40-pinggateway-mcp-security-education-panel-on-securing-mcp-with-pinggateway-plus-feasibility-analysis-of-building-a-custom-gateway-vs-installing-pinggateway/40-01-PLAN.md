  <action>
Create the multi-tab education panel with four tabs:

**Tab 1 — Overview: Why Gateway MCP?**
- MCP servers expose tools that perform real actions (transfers, account reads)
- Without a gateway: no rate limiting, no centralized auth enforcement, no audit trail
- PingGateway intercepts every WebSocket/HTTP call, validates bearer tokens, enforces scopes
- Key diagram: Client → PingGateway → MCP Server → Banking API

**Tab 2 — Architecture**
- ASCII/text diagram showing deployment topology
- PingGateway as reverse proxy in front of MCP server
- Token validation flow: PingGateway calls PingOne /as/introspect to validate tokens
- Scope enforcement: PingGateway checks required scopes per MCP tool
- WebSocket upgrade handling

**Tab 3 — Custom Gateway vs PingGateway**
Comparison table:

| Capability | Custom Gateway | PingGateway |
|---|---|---|
| Token validation | Build JWT/introspect logic | Built-in, auto-discovers JWKS |
| Scope enforcement | Manual route→scope mapping | Policy-driven, declarative |
| Rate limiting | Implement from scratch | Built-in, per-client policies |
| Audit logging | Custom logging pipeline | Integrated with PingOne audit |
| WebSocket support | Build upgrade handling | Native WebSocket proxy |
| Deployment | Docker/K8s, you maintain | Docker/K8s, Ping maintains |
| Time to production | Weeks | Hours |
| Cost | Engineering time | License fee |

Verdict: Custom gateway makes sense for learning; PingGateway for production.

**Tab 4 — Configuration Example**
Sample PingGateway route configuration (JSON/YAML) showing:
- Route to MCP server
- Token validation filter (introspect endpoint)
- Scope enforcement filter
- Rate limiting policy
- Audit handler

Follow existing education panel patterns (tabs, code blocks, diagrams).
