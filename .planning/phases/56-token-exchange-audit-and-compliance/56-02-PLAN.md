<objective>
Replace synthetic may_act claim injection with production-ready RFC 8693-compliant implementation.

**Current Problem:** 
- may_act claims created via feature flag `ff_inject_may_act` with incorrect format `{ client_id: "value" }`
- Only modifies in-memory token snapshot, not actual JWT
- Educational demo mode, not production-ready
- Violates RFC 8693 §4.2 specification (should be `{ sub: "value" }`)

**Solution:**
- Remove synthetic injection feature flag
- Implement proper subject claim validation in exchange responses
- Add configuration-based enablement for may_act support
- Ensure RFC 8693 compliance with real JWT-based claims
- Preserve backwards compatibility for single-exchange pattern

**Output:** 
- Production-ready may_act handling in `agentMcpTokenService.js`
- Configuration-driven implementation (no feature flags)
- RFC 8693-compliant claim format
- Full test coverage for both single and double exchange patterns
- All existing functionality preserved

**Why This Matters:**
RFC 8693 token exchange is core to our MCP agent architecture. Production deployment requires full compliance with RFC 8693 specification, particularly the may_act claim format which is used in nested delegation scenarios. Current synthetic injection violates the spec and prevents real compliance testing.
</objective>

## STRIDE Threat Register

| ID | Category | Component | Threat | Disposition | Mitigation |
|---|----------|-----------|--------|-------------|-----------|
| T-56-01 | Elevation of Privilege | may_act claim format | Synthetic { client_id } instead of { sub } allows forged delegation claims | Mitigate | Replace synthetic injection with real PingOne token policies; validate RFC 8693 format in implementation (Task 2) |
| T-56-02 | Spoofing | Subject claim | Exchanged token could have different subject than original user, enabling privilege escalation | Mitigate | Add explicit subject preservation validation with security audit event (Task 2, lines after 340) |
| T-56-03 | Denial of Service | Configuration | Feature flag maintenance creates attack surface for misconfiguration | Mitigate | Replace feature flags with standard configuration keys (Task 3: enableMayActSupport) |
| T-56-04 | Information Disclosure | Error Responses | Token exchange failures could expose internal PingOne api responses | Accept | Existing error handling already abstracts PingOne details; no new exposure |
| T-56-05 | Tampering | Token Claims | In-memory modification of claims before exchange could be intercepted | Mitigate | Remove synthetic injection that modifies userAccessTokenClaims; rely on PingOne to provide may_act (Task 2) |

**Mitigations Summary:**
- T-56-01: Synthetic injection removed, RFC 8693 validation added
- T-56-02: Subject preservation validator added with audit logging
- T-56-03: Feature flag replaced with typed configuration
- T-56-05: In-memory claim modification removed entirely
