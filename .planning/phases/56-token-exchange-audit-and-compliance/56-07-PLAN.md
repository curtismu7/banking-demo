**Section 1: Overview**
```markdown
# Two-Exchange Delegation Flow Guide

## What is Two-Exchange Delegation?
Two-exchange delegation (RFC 8693 §2.3) allows an AI Agent to act on behalf of a user by:
1. Getting an actor token (AI Agent client credentials)
2. Exchanging user token + agent actor → intermediate token with act.sub
3. Getting MCP actor token (MCP Exchanger client credentials)
4. Exchanging intermediate + MCP actor → final token with nested act claims

## Terminology
- **Subject Token**: Original user access token
- **Actor Token**: Credentials of entity acting on behalf (AI Agent or MCP Exchanger)
- **Exchanged Token**: Token resulting from exchange (contains act claim)
- **act Claim**: Nested structure indicating who acted: {act: {sub: "AI_AGENT", act: {sub: "MCP"}}}
- **Audience (aud)**: Resource that token can be used on (narrowed at each step)
```

**Section 2: Four-Step Flow**
```markdown
## The Four Exchange Steps

### Step 1: AI Agent Actor Token Acquisition
**Purpose**: Get credentials proving the AI Agent is authorized to act
**Input**: AI_AGENT_CLIENT_ID + AI_AGENT_CLIENT_SECRET
**Output**: agentActorToken (JWT with aud=agent_gateway_audience)
**PingOne App**: Super Banking AI Agent (Client Credentials grant)
**Error Conditions**:
- `invalid_client`: AI_AGENT_CLIENT_ID not recognized by PingOne
- `invalid_scope`: Agent gateway audience not valid for this client
- **Remediation**: [See Configuration Guide]

### Step 2: First Exchange (User Token + Agent Actor → Agent Exchanged Token)
**Purpose**: Prove user authorized agent to act, get token with act.sub=AI_AGENT_CLIENT_ID
**Input**: 
  - subject_token: User access token (contains may_act.sub = AI_AGENT_CLIENT_ID)
  - actor_token: agentActorToken from Step 1
  - audience: ai_agent_intermediate_audience
**Output**: agentExchangedToken (JWT with act.sub=AI_AGENT_CLIENT_ID)
**RFC 8693 Validation**: may_act.sub must equal actor_token.aud[0]
**Error Conditions**:
- `unauthorized`: may_act.sub doesn't match AI_AGENT_CLIENT_ID
- `invalid_grant`: Exchange failed (generic)
- **Remediation**: [See Configuration Guide → Common Issues]

### Step 3: MCP Actor Token Acquisition
**Purpose**: Get credentials proving MCP Exchanger is authorized
**Input**: AGENT_OAUTH_CLIENT_ID + AGENT_OAUTH_CLIENT_SECRET
**Output**: mcpActorToken (JWT with aud=mcp_gateway_audience)
**PingOne App**: MCP Token Exchanger (Client Credentials grant)
**Error Conditions**: Same as Step 1
- **Remediation**: [See Configuration Guide]

### Step 4: Second Exchange (Agent Exchanged + MCP Actor → Final Token with Nested Act)
**Purpose**: Get final token for MCP with nested act claims showing full delegation chain
**Input**:
  - subject_token: agentExchangedToken from Step 2
  - actor_token: mcpActorToken from Step 3
  - audience: mcp_resource_uri_two_exchange (Super Banking Resource Server)
**Output**: finalToken (JWT with nested act: {act: {sub: "MCP", act: {sub: "AI_AGENT"}}})
**RFC 8693 Validation**: act.sub from subject must equal actor_token.aud[0]
**Error Conditions**:
- `invalid_grant`: act.sub mismatch or Exchange #1 failed
- **Remediation**: [See Configuration Guide → Troubleshooting]
```

**Section 3: Nested Act Claims**
```markdown
## Understanding Nested Act Claims

### Single Exchange Resulting Token
```json
{
  "sub": "user-id",
  "aud": "mcp-server",
  "scope": "get_accounts:read transfer:execute",
  "iat": 1712702400
}
```

### Two-Exchange Resulting Token (Final)
```json
{
  "sub": "user-id",
  "aud": "mcp-resource-server",
  "scope": "get_accounts:read transfer:execute",
  "act": {
    "sub": "mcp-client-id",
    "act": {
      "sub": "ai-agent-client-id"
    }
  },
  "iat": 1712702400
}
```

### Reading the Act Chain
- **Original user**: "user-id"
- **MCP acting on behalf of original user**: "mcp-client-id"
- **AI Agent that triggered MCP**: "ai-agent-client-id"
- **Translation**: "AI Agent → MCP → User" (reverse order)
```

# Security Analysis: RFC 8693 Two-Exchange Delegation

## Trust Boundaries

```
┌─────────────┐
│   Browser   │  UNTRUSTED (same-origin only via CORS)
└──────┬──────┘
       │ HTTPS + PKCE
┌──────▼──────────────┐
│   BFF (Node.js)     │  SEMI-TRUSTED (our code, 3rd-party libs)
└──────┬──────────────┘
       │ mTLS + Client Credentials
┌──────▼──────────────┐
│  PingOne OAuth      │  TRUSTED (identity provider)
└─────────────────────┘

┌──────────────────┐
│  AI Agent        │  UNTRUSTED (untrusted AI, vendor-supplied)
└──────┬───────────┘
       │ Token Exchange
┌──────▼──────────────┐
│  MCP Server        │  SEMI-TRUSTED (MCP spec defines contract)
└──────┬──────────────┘
       │ Authenticate to Resource Server
┌──────▼──────────────┐
│  Resource Server   │  TRUSTED (validates act claims)
└─────────────────────┘
```

**Critical Boundary Crossings**:
1. **Browser → BFF**: User credentials + session state
2. **BFF → PingOne**: Client credentials for token exchange
3. **BFF → Agent**: exchanged token is released (must be narrowed)
4. **Agent → MCP**: Agent represents user + AI Agent
5. **MCP → Resource Server**: Final token with nested act claims (proves delegation chain)
```
