# Phase 71 — Plan 01: Feature & Sales Documentation

## Objective
Produce a polished, demo-ready `FEATURES.md` (and a matching in-app or PDF-ready
doc) that answers "what can this app do, and why should I care?" — written from a
sales / solutions-engineering perspective so anyone can pick it up, understand the
value, and use it to pitch the demo.

---

## Context

BX Finance is a full-stack AI banking demo that showcases PingOne's OAuth 2.0,
MFA, CIBA, and MCP (Model Context Protocol) capabilities in a realistic scenario.
It currently lacks a single, coherent "here is everything you can demo" document.
Prospective customers, SE teams, and partners should be able to scan this doc and
immediately see the full breadth of the platform without digging through code.

---

## Deliverables

1. **`docs/FEATURES.md`** — Markdown source (Gist-friendly, renders on GitHub).
2. **Sections required:**

   ### 1. Elevator Pitch (2–3 sentences)
   What BX Finance is, who it is for, and the single headline value.

   ### 2. Key Demo Scenarios
   Numbered use-cases a presales engineer would walk through:
   - UC-A: Standard login + banking (OAuth 2.0 / OIDC)
   - UC-B: Step-up MFA on high-value transactions (CIBA / OTP)
   - UC-C: AI agent (Banking Agent) — accounts, balances, transfers
   - UC-D: RFC 8693 Token Exchange — scoped MCP token before every tool call
   - UC-E: Family delegation — may_act / act claim chain
   - UC-F: Sensitive data access — PingOne Authorize / ACR gate
   - UC-G: Self-service provisioning — Management API worker token
   - UC-H: Admin / ops console — live token debug, verbose OAuth log

   ### 3. Feature Matrix (table)
   | Feature | Description | Where to Demo |
   With every significant capability (OAuth, CIBA, HITL, token exchange,
   MCP, ACR, may_act, step-up, delegation, self-service provisioning,
   multi-vertical, multi-layout, token chain visualizer, agent flow diagram,
   education panels, Postman collections, RFC 9728, …).

   ### 4. Integration Architecture
   Short prose + pointer to existing architecture diagrams (C4, flow diagrams).

   ### 5. Standards & RFCs (summary, link to Phase 72 RFC doc)
   One-liner per standard; "see RFC-STANDARDS.md for full coverage."

   ### 6. What Makes This Different
   Differentiators vs. generic OAuth demos:
   - MCP token exchange is real, not simulated
   - Token chain visualizer shows the full delegation chain live
   - Agent Flow Diagram shows per-step authorization in real time
   - Education panels explain the underlying RFCs in context
   - Multi-vertical (retail banking + HR workforce)
   - Self-service PingOne env setup via provisioning wizard

   ### 7. Quick Start / Pitch Checklist
   "5 things to show in a 20-minute demo" — ordered list.

---

## Implementation Notes

- Read existing `FEATURES.md`, `README.md`, `CHANGELOG.md`, all phase plans, and
  education panel content to extract capabilities already built.
- Do NOT invent features that don't exist — verify against code.
- Write at a **business / solutions-engineering level**, not developer level.
- Use tables, badges, and short bullet points — scannable in 2 minutes.
- Keep technical jargon minimal; explain acronyms on first use.
