  <name>Task 1: Create tokenIntrospectionService with PingOne integration</name>
  <files>banking_api_server/services/tokenIntrospectionService.js</files>
  <action>
Create a new service that validates tokens by calling PingOne's token introspection endpoint (RFC 7662). The service must:

1. **Function: validateToken(token: string): Promise<{valid: boolean, scopes: string[], sub: string, exp: number, aud: string, client_id: string}>**
   - Accept a Bearer token (JWT or opaque)
   - Call PingOne's introspection endpoint using the worker app credentials
   - Return token metadata including scopes (as an array, e.g., ["banking:read", "transactions:write"])
   - If token is invalid/expired: return {valid: false}
   
2. **Function: extractScopes(introspectionResponse: object): string[]**
   - Parse PingOne's scope field (may be space-separated string like "banking:read transactions:write")
   - Return array: ["banking:read", "transactions:write"]
   - Handle empty/missing scope field (return [])

3. **Caching (optional but recommended)**:
   - Optional in-memory token cache with TTL of 30 seconds
   - Key: hash of token; Value: introspection result
   - Avoids hammering PingOne on every tool call
   - BUT: respects token exp claim (don't cache beyond expiry)

4. **Error handling**:
   - If PingOne is unreachable: throw error with ctx for logging (don't silently fail)
   - Log the call: {token_hash: "...", scopes_returned: [...], timestamp: ...} for audit

5. **Environment variables** (document in code):
   - PINGONE_INTROSPECTION_ENDPOINT — PingOne introspection endpoint URL (e.g., "https://auth.pingone.com/v1/environments/{env_id}/oauth2/introspect")
   - PINGONE_WORKER_CLIENT_ID — Worker app client_id
   - PINGONE_WORKER_CLIENT_SECRET — Worker app secret
   - INTROSPECTION_CACHE_TTL — Cache TTL in seconds (default: 30)

Import existing patterns from `banking_api_server/services/oauthClientRegistry.js` for environment variable access and error handling.

   - Call tokenIntrospectionService.validateToken(token)
   - Return response per RFC 7662:
     ```json
     {
       "active": true,
       "scope": "banking:read transactions:write",
       "client_id": "external-client-123",
       "username": "claude@external.ai",
       "token_type": "Bearer",
       "exp": 1704067200,
       "iat": 1704067200,
       "sub": "user-uuid"
     }
     ```
   - If token is invalid: return {active: false}
   - If token validation fails (server error): return 500 with error message

2. **Error responses** (per RFC 6750):
   - Invalid token format: 400 Bad Request
   - Expired/invalid token: 200 OK with {active: false}
   - Server error during introspection: 500 with {error: "server_error"}

   - Attach to Express app in `banking_api_server/server.js` as `router.post('/api/introspect', validateTokenRoute)`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External Client → BFF | Client sends Bearer token; BFF validates at /api/introspect |
| BFF → PingOne | BFF uses worker app credentials to introspect tokens; communication must be HTTPS |
| MCP Gateway → /api/introspect | MCP gateway sends tokens from external clients; introspection result gates tool access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-01 | Spoofing | Token validation | Mitigate | Always validate tokens via PingOne introspection; no local token parsing for external clients. Worker app credentials in secure .env. |
| T-91-02 | Tampering | Introspection response | Mitigate | Validate HTTPS certificate chain on PingOne endpoint. Cache introspection result but respect exp claim to avoid using expired cached entries. |
| T-91-03 | Repudiation | Token usage | Mitigate | Log all introspection calls with token hash + result + timestamp for audit trail. |
| T-91-04 | Information Disclosure | Worker app secrets | Mitigate | Worker app credentials only in .env, not in code. Never log tokens, only token hashes. Introspection responses contain only non-sensitive metadata. |
| T-91-05 | Denial of Service | PingOne unreachable | Accept | If PingOne is unreachable, external clients cannot get tokens. This is acceptable; PingOne downtime impacts the entire auth system anyway. Implement timeout on PingOne calls (e.g., 5 seconds) to fail fast. |
| T-91-06 | Elevation of Privilege | Scope override | Mitigate | Scopes returned from PingOne introspection are the source of truth. MCP gateway (Phase 91 Plan 02) enforces these scopes; no client-side scope modification allowed. |

</threat_model>
