  <name>Task 1: Create MCP gateway token validator middleware</name>
  <files>banking_mcp_server/src/mcpGateway.js</files>
  <action>
Create a new module that validates Bearer tokens for MCP requests:

1. **Function: createTokenValidator(introspectUrl)**
   - Returns a middleware function for the MCP WebSocket server
   - On WebSocket connection, extracts Bearer token from upgrade request headers
   - Example header: "Authorization: Bearer eyJhbGciOiJIUzI1NiI..."
   - Calls POST /api/introspect with the token (calls BFF endpoint from Phase 91 Plan 01)
   - If token is valid: attach token metadata to connection context (ctx.token = {scopes, sub, exp, client_id})
   - If token is invalid/missing: reject connection with 403 Unauthorized

2. **Function: validateMCPRequest(ctx, toolName, requiredScopes)**
   - Check if the current connection has a valid token
   - Extract token scopes from ctx.token.scopes (array)
   - Check if requiredScopes (array) is a subset of token scopes
   - If allowed: return {allowed: true}
   - If denied: return {allowed: false, reason: "invalid_scopes"}
   - Example: Tool "getBalance" requires ["banking:read"]. Token has ["banking:read", "transactions:write"]. → allowed
   - Example: Tool "transfer" requires ["transactions:write"]. Token has ["banking:read"] only. → denied

3. **Tool scope requirements** (built-in to this phase):
   - Define a mapping of MCP tool name → required scope(s)
   - Example:
     ```
     const TOOL_SCOPES = {
       "getAccountBalance": ["banking:read"],
       "listTransactions": ["banking:read"],
       "validateTransactionLimit": ["transactions:validate"],
       "submitTransaction": ["transactions:write"],
       ...
     };
     ```
   - This mapping may come from Phase 90 scope/resource config or be hardcoded here
   - NOTE: Hardcoding is OK for Phase 91 Plan 02; Phase 94 can externalize this to configstore

4. **Error handling**:
   - If introspection endpoint is unreachable: reject connection with 500 Server Error
   - If token is malformed: reject with 400 Bad Request
   - If scope validation fails: reject with 403 invalid_scopes (MCP spec)
   - Log all validation decisions: {client_id, tool, scope_check, result, timestamp}

5. **Integration point**:
   - Attach validator to the existing MCP WebSocket server in `banking_mcp_server/index.js`
   - Example pseudocode:
     ```javascript
     const tokenValidator = createTokenValidator(process.env.BFF_INTROSPECT_URL);
     wss.on('connection', async (ws, req) => {
       const valid = await tokenValidator(req);
       if (!valid) { ws.close(403, 'Unauthorized'); return; }
       // ... existing MCP connection handler
     });
     ```

6. **Token refresh / expiration**:
   - MCP connections are persistent (WebSocket)
   - Token may expire while connection is open
   - On each tool call, re-validate the token (call introspect again)
   - OR: If connection lasts longer than token lifetime, close connection with 401 Unauthorized
   - Implementation: Option 1 is simpler for Phase 91; implement option 1

Import patterns from `banking_api_server/routes/transactions.js` for token validation and error handling.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External Client → MCP Gateway | Client sends Bearer token; gateway validates before accepting connection |
| MCP Gateway → /api/introspect | Gateway calls BFF to re-validate tokens; communication is server-to-server |
| MCP Gateway → Tool Handler | Only allowed tools (by scope) reach tool handler |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-91-02-01 | Spoofing | MCP connection auth | Mitigate | Always validate token before accepting connection. Re-validate on each tool call. No local token caching in MCP server. |
| T-91-02-02 | Tampering | Scope validation | Mitigate | Scope list is read-only from introspect response. Tool scope requirements are enforced at gateway entry point. |
| T-91-02-03 | Repudiation | Tool call audit | Mitigate | Log all tool calls with client_id + token_sub + tool_name + scope_check + timestamp. |
| T-91-02-04 | Information Disclosure | Token in logs | Mitigate | Never log the token itself, only token hash or metadata (client_id, sub, scopes). |
| T-91-02-05 | Denial of Service | Invalid tool requests | Mitigate | MCP server closes connections that send malformed requests. Rate limiting per client (future phase). |
| T-91-02-06 | Elevation of Privilege | Scope override | Mitigate | Scopes are read from introspect response, not from client request. Tool requirements are hardcoded/externalized but not user-configurable per request. |

</threat_model>
