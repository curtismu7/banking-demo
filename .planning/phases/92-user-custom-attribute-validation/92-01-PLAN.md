```json
    {

      "displayName": "Customer Tier",

      "description": "Customer tier for feature access",
      "example": "standard",
      "autoFixDefault": "standard"
    }
  ]
}
```

**Schema structure**:
- `required` — Attributes every user must have. Validator reports missing as errors.
- `optional` — Attributes that may be missing. Validator reports as warnings.
- Per attribute:
  - `name` — PingOne custom attribute name (e.g., "banking_account_number")
  - `type` — JSON type: "string", "number", "enum", "boolean"
  - `pattern` (for string) — Regex for validation (e.g., "^[0-9]{10}$" for 10 digits)
  - `allowedValues` (for enum) — List of valid values
  - `autoFixDefault` — What value to use if auto-fixing missing attribute
    - Special value: "AUTOMATED_ACCOUNT_SEQUENCE" → auto-increment from a counter

**Future extensibility**:
- Can add more attributes (e.g., "banking_daily_limit", "banking_api_key_count")
- Can add complex types (object, array)
- Versioning: If schema changes, increment schemaVersion

Load this schema into memory at BFF startup (cache it). Service layer reads from cache.

<task type="auto">
  <name>Task 2: Create userAttributeValidator service</name>
  <files>banking_api_server/services/userAttributeValidator.js</files>
  <action>
Create a service layer that validates and fixes user attributes:

1. **Function: validateUserAttributes(userId, userToken, schema)**
   - Load user's current attributes from PingOne
   - Compare against schema (required + optional)
   - Return validation result:
     ```json
     {
       "userId": "user-123",
       "valid": false,
       "issues": [
         {
           "attribute": "banking_account_number",
           "status": "missing",
           "required": true,
           "autoFixDefault": "AUTOMATED_ACCOUNT_SEQUENCE"
         },
         {
           "attribute": "banking_account_type",
           "status": "invalid",
           "currentValue": "unknown",
           "allowedValues": ["checking", "savings"],
           "required": true
         }
       ],
       "attributeCount": {
         "present": 2,
         "missing": 2,
         "invalid": 1
       }
     }
     ```

2. **Function: fixUserAttributes(userId, adminToken, schema)**
   - For each attribute in issues that has autoFixDefault:
   - Call PingOne Management API to update the attribute
   - Handle "AUTOMATED_ACCOUNT_SEQUENCE" by:
     - Reading current account counter from configStore or in-memory
     - Incrementing counter
     - Assigning next sequential account number
   - Return fix result:
     ```json
     {
       "userId": "user-123",
       "fixed": ["banking_account_number", "banking_account_type"],
       "skipped": ["banking_account_type_invalid"],
       "errors": []
     }
     ```

3. **Function: getSchem()**
   - Load and return the userAttributeSchema.json (cache at startup for performance)
   - If schema not loaded: load from file synchronously at startup

4. **PingOne API calls**:
   - To read attributes: GET /v1/environments/{env_id}/users/{user_id} → extract custom:banking_* attributes
   - To write attributes: PATCH /v1/environments/{env_id}/users/{user_id} with {"customAttributes": {...}}
   - Use worker app credentials for these calls (already available from Phase 91)

5. **Error handling**:
   - If user not found: return {valid: null, error: "user_not_found"}
   - If PingOne API error: handle gracefully, log, return error in response
   - If attribute pattern validation fails: mark as "invalid" in issues list

6. **Logging**:
   - Log validation calls: {user_id, attribute_count_present, attribute_count_missing, timestamp}
   - Log fix calls: {user_id, attributes_fixed, timestamp}

Import patterns from `banking_api_server/services/oauthClientRegistry.js` for PingOne API calls and error handling.

<task type="auto">
  <name>Task 3: Create admin endpoint for user attribute validation</name>
  <files>banking_api_server/routes/admin-validate-attributes.js</files>
  <action>
Create an Express route for validation and fixing user attributes:

1. **GET /api/admin/user-attributes**
   - Returns the current userAttributeSchema
   - Admin can use this to understand what attributes are required
   - Response:
     ```json
     {
       "schema": { ... },
       "description": "Required and optional user attributes for the banking demo"
     }
     ```

2. **POST /api/admin/user-attributes/validate**
   - Admin provides user ID (or uses current session user)
   - Request: `{userId: "user-123"}` OR uses bearer token to infer userId
   - Calls userAttributeValidator.validateUserAttributes()
   - Response: Validation result (see Task 2 format)

3. **POST /api/admin/user-attributes/validate-and-fix**
   - Admin provides user ID
   - Calls userAttributeValidator.validateUserAttributes() first
   - If issues found, calls userAttributeValidator.fixUserAttributes()
   - Returns both validation and fix results in one response
   - Response:
     ```json
     {
       "validation": { ... },
       "fix": { ... },
       "message": "Fixed 2 attributes, 1 remain invalid"
     }
     ```

4. **Admin authentication**:
   - Require bearer token with admin scope (e.g., "admin:read" for validate, "admin:write" for fix)
   - Validate token via introspect endpoint (from Phase 91 Plan 01)
   - Non-admin users calling these endpoints get 403 Forbidden

5. **Error handling**:
   - Invalid userId: 400 Bad Request
   - User not found in PingOne: 404 Not Found
   - Not authenticated: 401 Unauthorized
   - Not admin: 403 Forbidden
   - PingOne API error: 500 Internal Server Error

6. **Integration into server.js**:
   - Import and mount router: `app.use(routes.adminAttributes);`
   - OR add routes to existing admin.js file if it exists

Patterns from `banking_api_server/routes/transactions.js` for token validation and error middleware.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin → /api/admin/user-attributes | Admin provides user ID; BFF validates admin token before allowing validation/fix |
| BFF → PingOne Management API | BFF uses worker app credentials to read/write user attributes |
| Validator → Schema file | Schema is read-only; defines what attributes are required and what defaults to use |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-92-01 | Spoofing | Admin validation endpoint | Mitigate | Require admin token with "admin:read" or "admin:write" scope. Validate token via introspect endpoint. |
| T-92-02 | Tampering | User attributes | Mitigate | PingOne API calls use HTTPS. Worker app credentials are secure. Attributes are validated against schema (regex patterns). |
| T-92-03 | Repudiation | Attribute fix audit | Mitigate | Log all fix operations: user_id + attributes_changed + old_value → new_value + admin_token_sub + timestamp. |
| T-92-04 | Information Disclosure | User attribute data | Mitigate | Validator responses contain only attribute names and values (not PII beyond what's already user-facing). Worker app secrets stay in .env. |
| T-92-05 | Denial of Service | Validation endpoint | Accept | If PingOne is down, validation fails. This is acceptable; the entire auth system depends on PingOne anyway. Implement timeout on PingOne calls. |
| T-92-06 | Elevation of Privilege | Auto-fix defaults | Mitigate | Defaults are hardcoded in schema (not user-configurable). Admin cannot override defaults; can only accept or reject fix operation. |

</threat_model>
