  <name>Task 1: Define agent operation event format and BFF route</name>
  <files>banking_api_server/routes/agent-events.js</files>
  <action>
Create a BFF route that serves agent operation events to the UI:

1. **Event format** (standardized):
   ```json
   {
     "operationId": "op-123",
     "userId": "user-456",
     "type": "operation",
     "event": "operation_started" | "step_started" | "step_completed" | "tool_call" | "tool_result" | "operation_complete" | "error",
     "step": 2,
     "stepName": "Validating transaction limits",
     "toolName": "validateTransactionLimit",
     "scope": "transactions:validate",
     "dataAccessed": "Account ending in 1234",
     "result": { "isValid": true, "limitRemaining": 450 },
     "error": null,
     "timestamp": "2026-04-08T12:00:00Z"
   }
   ```

2. **POST /api/agent-operations/{operationId}/start**
   - Agent/BFF creates a new operation with operationId
   - Stores operation metadata in a temporary store (in-memory Map, Redis, or session)
   - Returns {operationId, started: true}
   - Called by agent before starting tool calls

3. **GET /api/agent-operations/{operationId}** (polling option)
   - Returns current operation status + list of events so far
   - Response:
     ```json
     {
       "operationId": "op-123",
       "status": "in_progress" | "complete" | "error",
       "events": [ { ... }, { ... }, ... ],
       "currentStep": 2,
       "estimatedRemaining": "30 seconds"
     }
     ```
   - Polling option: UI calls this every 500ms to get latest status

4. **GET /api/agent-operations/{operationId}/events** (SSE option)
   - Returns Server-Sent Events stream
   - One event per line: `data: {...}\n\n`
   - Can be more efficient than polling, but requires WebSocket or EventSource support
   - Implementation: Optional for Phase 93 Plan 01; polling is sufficient

5. **POST /api/agent-operations/{operationId}/event**
   - Agent or middleware posts a new event to the operation
   - Request: Event object (see format above)
   - Response: {eventId, recorded: true}
   - Called by agent after each tool call completes

6. **Operation storage**:
   - Store active operations in memory (Map) or session store (Upstash)
   - Key: operationId; Value: {metadata, events: []}
   - Expire after 1 hour (operation complete or old)
   - Do NOT store in database (too much I/O)

7. **Authentication**:
   - Require bearer token
   - Allow any authenticated user to read their own operations (userId match)
   - Allow agent (with special scope like "agent:debug") to post events

Import patterns from `banking_api_server/routes/transactions.js` for token validation.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User → Agent Delegation | User authorizes agent via OAuth consent; agent receives delegation token with `act` claim |
| Agent → MCP Tool Calls | Each tool call includes agent token with `act` claim; MCP server validates scope |
| UI → Operation Events | UI polls /api/agent-operations endpoint for events; events are user-specific (owner check) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-93-01 | Spoofing | Operation event stream | Mitigate | Events are served by authenticated endpoint (bearer token required). UI can only request events for own operations (userId match). |
| T-93-02 | Tampering | Operation display | Accept | If someone tampers with UI to show fake operations, the actual audit trail (server logs) is unaffected. This is UI cosmetics fraud, not actual fraud. |
| T-93-03 | Repudiation | Operation audit | Mitigate | Server logs all operation events with full metadata (user, timestamp, tool, scopes). Audit trail is server-side and immutable. |
| T-93-04 | Information Disclosure | Operation events | Mitigate | Events contain only non-sensitive metadata (tool names, scopes). Never include account balances, transaction amounts, or PII in event stream. |
| T-93-05 | Denial of Service | Event polling | Mitigate | Polling endpoint has rate limiting per user (future phase). Event stream expires after 1 hour. |
| T-93-06 | Elevation of Privilege | Delegation | Mitigate | User sees exactly what delegation they authorized. Education modal explains scopes clearly. User can revoke access immediately in Settings. |

</threat_model>
