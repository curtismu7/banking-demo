## RFC 8693 §4.1-4.4 Specification Mapping

### Definition: may_act vs act Claims

**RFC 8693 §4.1 (may_act - User Authorization)**
- **On:** User's access token (T1 - before exchange)
- **Meaning:** User is authorized to delegate their authority to a specific client
- **Structure:** `{ client_id: "..." }` — identifies which BFF client is authorized
- **Scope:** PingOne configures via user attribute during login; part of T1 token
- **Validation:** Middleware checks user's `may_act.client_id` matches the BFF's client_id

**RFC 8693 §4.1-4.2 (act - Actor Proof)**
- **On:** Output token (T2, T3, and all downstream tokens)
- **Meaning:** Proof that a specific actor (identified by `sub` field) is acting on behalf of the original user
- **Structure (1-exchange):** `{ sub: "<bff-client-id>" }` — BFF is the actor
- **Structure (2-exchange):** `{ sub: "<mcp-client-id>", act: { sub: "<agent-client-id>" } }` — nested chain
- **Scope:** Generated by PingOne during token exchange; preserved in all downstream tokens
- **Validation:** Middleware validates `act.sub` identifies valid actor

**RFC 8693 §4.4 (Nested act - Delegation Chain)**
- **Purpose:** Audit trail showing full delegation chain (who delegated to whom)
- **Structure:** Each `act` claim contains nested `act` of the delegator
- **Example:** 
  ```
  MCP Token sub=mcp-client-id:
    act: {
      sub: agent-client-id,  // agent is acting...
      act: {                 // ...on behalf of user (implicit in act chain)
        sub: user-guid
      }
    }
  ```
- **Requirement:** Full chain must be preserved through all token exchanges

- [banking_api_server/services/delegationClaimsService.js](banking_api_server/services/delegationClaimsService.js#L141-L200): `validateMayActStructure()` checks required field: **sub**
- Wait - this seems wrong! Let me re-examine...

Actually, looking at line 32 of delegationClaimsService.js:

```javascript
  required_fields: ['sub'],
  optional_fields: ['client_id', 'actor', 'may_act']

```

This says `sub` is required in `may_act`, but RFC 8693 and our documentation show `client_id` as the identifier. This might be a flexible validator allowing either `sub` OR `client_id` to identify the delegator. Let me check the actual validation logic...

Looking at line 162-168:
```javascript
for (const field of structure.required_fields) {
  if (!mayAct[field]) {
    validation.valid = false;
    validation.errors.push(`Missing required field in may_act: ${field}`);
  }
}
```

This checks `mayAct[field]` for each required field. But if `required_fields: ['sub']`, then it's requiring `may_act.sub` to exist. However, RFC 8693 and our code generate `may_act: { client_id: "..." }`. This is a **POTENTIAL ISSUE**.

## Discovered Issue: may_act Validation Rule

**Issue:** DELEGATION_RULES (line 32) specifies `may_act_structure.required_fields: ['sub']`

**But:** Our documentation and RFC 8693 show `may_act: { client_id: "..." }` (no sub field)

**Expected:** `may_act_structure.required_fields: ['client_id']` OR flexible `['client_id', 'sub']` (either one required)

**Impact:** If PingOne strictly returns `may_act: { client_id: "..." }`, the validation will FAIL because `sub` is missing.

**Resolution:** Need to verify:
1. What does PingOne actually return for may_act?
2. Should validation allow `client_id` as identifier instead of/in addition to `sub`?

### Blocking Issue: may_act Field Validation

**Severity:** ⚠️ RESOLVED (Commit 95275de)  
**Status:** ✅ Fixed

The validation rule for `may_act` was incorrectly requiring `sub` field instead of `client_id`:
- **Issue Found:** DELEGATION_RULES.user_token.may_act_structure.required_fields was `['sub']`
- **RFC 8693 §4.1:** Specifies `may_act: { client_id: "..." }` (client_id is the identifier)
- **Our docs:** Specified `may_act: { client_id: "bff-admin-client-id" }`
- **Validation code (BEFORE):** Required `may_act.sub` (incorrect)

**Fix Applied (Commit 95275de):**
- Updated `required_fields: ['client_id']` (RFC-compliant)
- Added validation for `client_id` field (string, non-empty)
- Kept optional `sub` field validation for backward compatibility
- Added RFC reference comments

**Impact:** Tokens from PingOne with `may_act: { client_id: "..." }` now pass validation correctly.
